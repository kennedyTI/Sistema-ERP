# Sistema ERP v2.4.0 - Status e alertas de impressoras

## Objetivo

Registrar a entrega da fase de Status/Alertas do módulo Impressoras, promovida
para `main`, consolidando a consulta operacional de máquinas ativas com alertas
atuais, histórico consultivo e execução agendada por Redis/Celery.

## Principais entregas

- status operacional consultivo para máquinas ativas;
- conectividade automática em ciclo de 60 segundos;
- coleta e sincronização de alertas operacionais em ciclo agendado;
- persistência de alertas atuais e histórico de alertas;
- regras de alertas e OIDs SNMP administrados pelo Django Admin;
- credenciais de coleta HTML protegidas e fora do código-fonte;
- fallback HTML/HTTP por modelo quando SNMP não retorna alerta útil;
- fallback IPP para HP MFP-4303;
- priorização por severidade e alternância de alertas equivalentes no frontend;
- modal de Status com detalhes operacionais e logs das últimas 24 horas;
- Docker com proxy HTTPS local, Redis, Celery Worker e Celery Beat validados.

## Validações executadas

- `python -m compileall -q backend`;

- `python manage.py check`: nenhuma inconsistência;
- `npm audit`: 0 vulnerabilidades;
- `npm run build`: build aprovado;
- Docker Compose: serviços ativos e migrations finalizadas com exit code 0;
- Redis, Celery Worker, Celery Beat e proxy HTTPS ativos;
- HTTPS local: login, início, Admin, autenticação, máquinas e status;
- API real: login, `/api/v2/auth/me` e status de impressoras;
- auditoria de logs sem HTML bruto, JS bruto, tokens ou payload técnico exposto;
- verificação de que arquivos sensíveis, certificados reais e dados locais não
  foram rastreados.

## Decisões técnicas

- a tela Status exibe somente máquinas ativas;
- status e logs operacionais continuam somente leitura por Admin/API manual;
- a cascata de coleta de alertas fica concentrada nos services do módulo
  Impressoras, preservando contratos de frontend e API;
- alertas simultâneos de mesma severidade podem alternar visualmente no
  frontend;
- quando há severidades diferentes, prevalece a mais crítica;
- fallbacks HTML/HTTP e IPP são específicos por modelo e não ampliam coleta rica;
- HTML bruto, JS bruto e segredos não são persistidos.

## Limitações não bloqueantes

- o build frontend mantém o aviso conhecido de chunks acima de 500 kB.
